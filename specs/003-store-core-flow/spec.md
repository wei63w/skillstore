## User Scenarios & Testing *(mandatory)*

### User Story 1 - 用户注册登录并获得角色身份 (Priority: P1)

作为买家、创作者或管理员，我需要能注册演示账号并登录获得角色身份，以便后续流程能按权限区分操作。

**Why this priority**: 所有下单、上传和审核流程都依赖身份与角色边界。

**Independent Test**: 用户可以注册三类角色账号，登录后看到 token、角色和用户信息；错误密码或重复用户名会被拒绝。

**Acceptance Scenarios**:

1. **Given** 用户提供唯一用户名、邮箱、密码和角色，**When** 提交注册，**Then** 系统创建账号并返回不含密码明文的用户信息。
2. **Given** 用户已注册，**When** 使用正确用户名和密码登录，**Then** 系统返回可用于后续请求的演示 token。
3. **Given** 用户使用错误密码登录，**When** 提交登录，**Then** 系统返回明确失败信息且不返回 token。

---

### User Story 2 - 创作者上传 Skill 并提交审核 (Priority: P2)

作为创作者，我需要上传 Skill 包、填写商品信息并提交审核，以便平台审核后让买家购买。

**Why this priority**: Skill 商品供给是商城交易闭环的前置条件。

**Independent Test**: 创作者登录后可上传受限类型和大小的 Skill 文件，创建 Skill 草稿并提交为待审核状态；非法文件类型或缺少创作者身份会被拒绝。

**Acceptance Scenarios**:

1. **Given** 创作者已登录，**When** 上传合法 Skill 包并填写名称、摘要、分类、定价，**Then** 系统生成 Skill 记录并标记为待审核。
2. **Given** 上传文件后缀不在允许范围内，**When** 创作者提交上传，**Then** 系统拒绝并说明原因。
3. **Given** 买家尝试上传 Skill，**When** 发起上传请求，**Then** 系统拒绝越权操作。

---

### User Story 3 - 管理员审核上架 Skill (Priority: P3)

作为平台管理员，我需要审核待上架 Skill，通过或拒绝并记录审计日志，以便控制商品安全上架。

**Why this priority**: 平台审核是 Skill 安全和商城可信交易的关键门禁。

**Independent Test**: 管理员可以查看待审核 Skill、通过审核使其出现在买家可购买列表，或拒绝并记录理由；非管理员不能审核。

**Acceptance Scenarios**:

1. **Given** 有待审核 Skill，**When** 管理员通过审核，**Then** Skill 状态变为已上架并写入审计日志。
2. **Given** 有待审核 Skill，**When** 管理员拒绝审核并填写理由，**Then** Skill 状态变为已拒绝并写入审计日志。
3. **Given** 创作者尝试审核，**When** 发起审核请求，**Then** 系统拒绝越权操作。

---

### User Story 4 - 买家下单并完成模拟支付 (Priority: P4)

作为买家，我需要购买已上架 Skill 并完成模拟支付，以便获得已购 Skill 的下载或试用授权。

**Why this priority**: 下单和支付模拟构成商城收入闭环的核心。

**Independent Test**: 买家可对已上架 Skill 创建订单，模拟支付成功后订单变为已支付，买家可查看已购 Skill；未审核商品不能下单。

**Acceptance Scenarios**:

1. **Given** Skill 已通过审核，**When** 买家创建订单，**Then** 系统生成待支付订单。
2. **Given** 订单待支付，**When** 买家选择模拟支付成功，**Then** 订单变为已支付并产生购买授权。
3. **Given** Skill 未通过审核，**When** 买家尝试下单，**Then** 系统拒绝购买。

### Edge Cases

- 重复用户名、弱密码、缺少角色、非法角色注册必须被拒绝。
- 上传文件必须限制后缀、大小和空文件；本阶段不执行、不解压上传内容。
- 非创作者上传、非管理员审核、非买家下单或支付必须被拒绝。
- 已拒绝或待审核 Skill 不能出现在买家可购买列表。
- 同一买家重复购买同一 Skill 时，系统必须返回已有授权或阻止重复支付。
- 模拟支付失败时订单保持待支付或失败状态，不生成购买授权。
- 所有敏感动作必须记录审计日志，包括注册、上传、审核、下单和支付模拟。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持买家、创作者、管理员三类演示账号注册。
- **FR-002**: 系统 MUST 使用加盐哈希保存密码，任何响应都 MUST NOT 返回密码明文或密码哈希。
- **FR-003**: 系统 MUST 支持用户名密码登录并返回演示 token、用户 ID、用户名和角色。
- **FR-004**: 系统 MUST 基于 token 识别当前用户角色，并对上传、审核、下单、支付执行角色校验。
- **FR-005**: 系统 MUST 支持创作者上传 Skill 包元数据和受限文件内容，允许后缀为 `.zip`、`.json`、`.txt`，单文件大小不超过 1MB。
- **FR-006**: 系统 MUST 支持创作者创建 Skill，字段包括名称、摘要、分类、定价模式和价格，初始状态为待审核。
- **FR-007**: 系统 MUST 支持管理员查询待审核 Skill，并通过或拒绝审核。
- **FR-008**: 系统 MUST 仅向买家展示已通过审核的 Skill。
- **FR-009**: 系统 MUST 支持买家对已通过审核 Skill 创建订单，订单初始状态为待支付。
- **FR-010**: 系统 MUST 支持模拟支付成功和失败；成功后订单变为已支付并生成购买授权，失败不得生成授权。
- **FR-011**: 系统 MUST 支持买家查询已购 Skill，并为已支付订单返回下载授权占位信息。
- **FR-012**: 系统 MUST 为注册、上传、审核、下单和支付模拟记录审计日志。
- **FR-013**: 系统 MUST 提供前端演示入口，让用户可在买家、创作者、管理员区域完成对应闭环操作。
- **FR-014**: 系统 MUST 以统一响应体返回成功和失败，并提供清晰错误消息。

### Constitution Requirements *(mandatory)*

- **CR-001**: 本功能 MUST 持久记录 Spec Kit 工作流产物、实现任务状态、测试日志、构建日志和 GitHub 提交。
- **CR-002**: 本功能 MUST 明确支付、隐私、上传、权限和安全审核的人工风险边界；本阶段不得接入真实支付或真实银行卡。
- **CR-003**: 本功能 MUST 保持商城代码在 `skill-store/` 下，不得改动 Harness 核心业务边界。
- **CR-004**: 本功能 MUST 覆盖认证、授权、输入校验、文件限制、密钥处理、安全扫描和依赖扫描要求。
- **CR-005**: 本功能 MUST 包含后端集成测试、服务单元测试、前端核心流程测试和依赖审计。
- **CR-006**: 本功能 MUST 保持演示交互在 10 分钟内可完成，常规页面和接口在本地开发环境中即时反馈。
- **CR-007**: 本功能 MUST 更新 API、数据库、安全、部署和 quickstart 文档。
- **CR-008**: 本功能 MUST 更新 README 开发记录。
- **CR-009**: 本功能 MUST 以独立 GitHub 提交完成，提交信息遵循 `[模块] 动作：内容`。
- **CR-010**: 本功能 MUST 严格执行 `specify`、`clarify`、`checklist`、`plan`、`tasks`、`implement` 顺序。

### Key Entities

- **用户账号**: 表示买家、创作者、管理员身份，包含用户名、邮箱、密码哈希、角色和状态。
- **会话令牌**: 表示登录后的演示访问凭证，关联用户和过期时间。
- **Skill 商品**: 表示创作者上传的商品，包含文件元数据、定价、审核状态和创作者。
- **审核记录**: 表示管理员对 Skill 的通过或拒绝动作和理由。
- **订单**: 表示买家购买 Skill 的交易单，包含订单号、金额、支付状态和买家。
- **购买授权**: 表示支付成功后买家对 Skill 的下载或试用权。
- **审计日志**: 表示敏感操作的行为记录。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 演示用户可在 10 分钟内完成注册、登录、上传、审核、下单、模拟支付和查看已购 Skill。
- **SC-002**: 100% 的敏感操作按角色限制，非授权角色无法上传、审核、下单或支付他人订单。
- **SC-003**: 100% 的上传请求执行后缀、大小和空文件校验。
- **SC-004**: 100% 的模拟支付成功订单生成购买授权，失败支付不生成授权。
- **SC-005**: 100% 的注册、上传、审核、下单、支付模拟动作产生审计日志。
- **SC-006**: 后端和前端自动化测试覆盖闭环主流程和关键失败路径。
- **SC-007**: README 开发记录包含日期、类型、范围、验证结果和 GitHub 状态。
- **SC-008**: Spec Kit 产物覆盖 specify、clarify、checklist、plan、tasks 和 implement 执行证据。

## Assumptions

- 本阶段是本地演示闭环，不接入真实支付渠道、真实银行数据、真实云部署或恶意代码沙箱执行。
- 演示 token 可用于本地 API 鉴权；生产级 JWT、刷新令牌和会话吊销后续单独增强。
- 数据可使用演示仓储支撑端到端闭环；生产 MySQL 持久化映射和迁移可在后续功能加强。
- 上传文件只用于演示元数据和受限内容保存，不执行、不解压、不加载用户上传脚本。
