## User Scenarios & Testing *(mandatory)*

### User Story 1 - 查看 Harness 项目入口 (Priority: P1)

作为项目维护者，我需要看到 Agent Dev Harness 的工程入口、模块边界、文档入口和
启动/验证指引，以便确认后续规划、编码、测试、构建部署 Agent 可以在统一骨架上继续
演进。

**Why this priority**: 没有清晰工程入口和边界，后续任何 Agent 能力都会散落在无约束
位置，无法满足可复现交付和分层解耦原则。

**Independent Test**: 从仓库根目录阅读 README 和 Harness 入口文档，能够在 10 分钟内
识别 Harness 的目标、模块边界、运行占位、测试占位、日志/状态产物位置和后续扩展点。

**Acceptance Scenarios**:

1. **Given** 新开发者首次打开仓库，**When** 阅读 README 和 Harness 入口文档，
   **Then** 能明确 Agent Dev Harness 与 OpenClaw Skill 商店业务应用是独立边界。
2. **Given** 后续要添加规划 Agent 或测试 Agent，**When** 查看工程骨架，
   **Then** 能找到对应模块位置、文档要求、测试位置和日志/状态产物约定。

---

### User Story 2 - 验证长运行任务基础约定 (Priority: P2)

作为 Harness 开发者，我需要骨架预留任务状态、执行日志、上下文摘要、人工确认和工具
调用记录的标准位置与说明，以便后续功能可以直接接入可观测和断点续跑能力。

**Why this priority**: 长运行 Agent Harness 的核心价值依赖可观测、可恢复和可审计的
执行记录，骨架阶段必须先固定这些约定。

**Independent Test**: 检查骨架文档和占位结构，确认每类运行产物都有明确用途说明、
命名约定和保留策略说明。

**Acceptance Scenarios**:

1. **Given** 一个长运行任务被规划，**When** 需要保存中间结果，
   **Then** 骨架说明能指向任务状态、上下文摘要和阶段产物的标准位置。
2. **Given** 一个操作需要人工确认，**When** 开发者查看交互约定，
   **Then** 能看到待确认内容、风险等级、可选方案和确认结果的记录要求。

---

### User Story 3 - 准备质量门禁与增量提交流程 (Priority: P3)

作为项目负责人，我需要骨架包含测试、扫描、构建、部署和 README 开发记录的占位流程，
以便后续每个小功能都能按同一标准验证、记录并提交到 GitHub。

**Why this priority**: 工程骨架不仅要能放代码，还要能承载项目宪章中的自动化闭环、
中文文档和增量提交要求。

**Independent Test**: 检查骨架后，能够列出后续小功能完成时需要更新的文档、需要运行
的验证项、需要记录的 README 字段和需要执行的 GitHub 提交流程。

**Acceptance Scenarios**:

1. **Given** 一个小功能完成，**When** 开发者准备收尾，
   **Then** 骨架提供 README 开发记录、验证结果和 GitHub 提交状态的记录位置。
2. **Given** 骨架尚未包含真实业务代码，**When** 执行质量门禁，
   **Then** 能清楚区分当前可验证项和后续功能必须补齐的测试、扫描、构建、部署项。

### Edge Cases

- 当仓库中已存在与 Harness 相关的目录或文档时，骨架初始化必须复用或合并现有内容，
  不得覆盖用户已有工作。
- 当当前阶段尚无可运行代码时，测试、扫描、构建和部署必须以明确的占位说明记录，不得
  假装已经完成真实执行。
- 当 GitHub 远程仓库不可用或推送失败时，README 开发记录必须记录阻塞原因和后续补交
  要求。
- 当 Harness 与 Skill 商店目录同时出现时，文档必须明确两者边界，避免业务代码进入
  Harness 核心模块。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 创建 Agent Dev Harness 的工程入口说明，明确项目目标、使用者、
  模块边界和后续扩展方式。
- **FR-002**: 系统 MUST 定义 Harness 与 OpenClaw Skill 商店业务应用的隔离边界，说明
  Harness 不承载商店业务规则。
- **FR-003**: 系统 MUST 为规划、编码、测试、构建部署、评审、工具插件、任务状态、
  上下文管理、人工确认和运行产物提供清晰的骨架位置或占位说明。
- **FR-004**: 系统 MUST 定义长运行任务的状态、日志、上下文摘要、工具调用记录和阶段
  产物的命名与用途约定。
- **FR-005**: 系统 MUST 提供中文优先的文档入口，并说明后续需求、计划、任务、测试、
  部署和运维文档的默认语言规则。
- **FR-006**: 系统 MUST 提供质量门禁占位，覆盖单元测试、集成测试、适用时的 E2E、
  安全扫描、依赖扫描、构建、部署、冒烟验证和回滚验证。
- **FR-007**: 系统 MUST 更新 README 开发记录，记录本次骨架初始化的日期、类型、范围、
  说明、验证结果和 GitHub 状态。
- **FR-008**: 系统 MUST 为后续每个小功能的 README 记录和 GitHub 增量提交提供明确流程
  说明。
- **FR-009**: 系统 MUST 避免引入真实密钥、真实服务器地址、管理员凭据或线上破坏性
  操作。
- **FR-010**: 系统 MUST 在没有真实运行能力的地方明确标注为待实现或占位，不得给出
  误导性的已完成状态。

### Constitution Requirements *(mandatory)*

- **CR-001**: 本功能 MUST 定义 Agent 操作、文档生成、验证动作和后续部署动作应如何
  记录为持久、可回放证据。
- **CR-002**: 本功能 MUST 标识高风险基础设施、权限、凭据、支付、隐私、安全和无法
  自动修复发布阻断问题的人工确认门禁。
- **CR-003**: 本功能 MUST 保持 Agent Dev Harness 与 OpenClaw Skill 商店业务应用分离。
- **CR-004**: 本功能 MUST 明确当前阶段不引入真实凭据、线上权限、文件执行或支付配置，
  并为后续安全扫描预留入口。
- **CR-005**: 本功能 MUST 说明当前骨架阶段可验证的文档检查，以及后续代码阶段必须
  补齐的单元、集成、E2E、SAST 和依赖扫描。
- **CR-006**: 本功能 MUST 声明 Harness 相关性能基线，包括断点恢复、并发流水线和单
  模块生成耗时在后续实现阶段必须被验证。
- **CR-007**: 本功能 MUST 声明后续构建、部署、回滚和运维文档输出要求。
- **CR-008**: 本功能 MUST 更新 README.md 开发记录，记录本次规格创建和后续骨架实现
  的变更追踪方式。
- **CR-009**: 本功能 MUST 提交到 GitHub，提交范围为 Agent Dev Harness 骨架规格初始化，
  提交信息遵循 `[模块] 动作：内容` 格式。

### Key Entities *(include if feature involves data)*

- **Harness 工程骨架**: 表示 Agent Dev Harness 的顶层工程入口、模块边界、文档入口、
  验证入口和运行产物约定。
- **Agent 模块占位**: 表示规划、编码、测试、构建部署和评审等 Agent 能力的未来扩展
  位置与责任边界。
- **运行产物约定**: 表示任务状态、结构化日志、上下文摘要、工具调用记录、测试报告和
  部署记录等可观测数据的分类与用途。
- **人工确认门禁**: 表示需要人类确认的风险事件、问题格式、可选方案和确认结果记录。
- **开发记录**: 表示 README 中按日期记录的小功能、开发修改、修复、验证结果和 GitHub
  提交状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新开发者在 10 分钟内能够从 README 和 Harness 入口文档识别项目目标、
  模块边界和后续扩展点。
- **SC-002**: 100% 的 Harness 核心责任域都有明确的骨架位置或占位说明，包括规划、
  编码、测试、构建部署、评审、工具、状态、上下文和人工确认。
- **SC-003**: 100% 的运行产物类别都有说明，包括任务状态、日志、上下文摘要、工具
  调用记录、测试报告、构建记录和部署记录。
- **SC-004**: 本次变更的 README 开发记录包含日期、类型、范围、说明、验证结果和
  GitHub 状态。
- **SC-005**: 规格质量清单全部通过，且没有待澄清标记。
- **SC-006**: 本次规格初始化以一个独立 GitHub 提交完成，提交信息符合项目规范。

## Assumptions

- 本次 `/speckit-specify` 只创建功能规格、质量清单和项目记录，不直接实现 Harness
  代码骨架。
- 工程骨架实现将在后续 `/speckit-plan`、`/speckit-tasks` 和实现阶段继续推进。
- 当前 GitHub 远程仓库可用于提交本次规格初始化记录。
- 文档默认使用中文，必要英文术语保留原文以避免误解。
